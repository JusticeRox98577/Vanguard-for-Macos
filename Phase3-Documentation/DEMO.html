01 — The Problem

Why VALORANT can't ship on macOS

Kernel-level anti-cheat is today's bar on Windows. It is also the single reason titles like VALORANT cannot ship on macOS.

vgc.sys, Easy Anti-Cheat, and BattlEye all load a ring-0 kernel driver at boot. Apple effectively prohibits third-party kernel extensions: the kext pipeline is deprecated, and loading one requires explicit Apple notarization, user approval, and a full reboot — a user-experience non-starter for a competitive game.

The result is an entire platform of players excluded by an implementation detail of the security model, not by the security requirement itself. Millions of Mac users who want to play VALORANT cannot — not because the Mac is less secure, but because the current approach assumes a kernel driver is the only path.

The real question
How do you protect the client on a platform with no kernel driver? This project is one concrete, auditable answer — built and tested.

02 — The Thesis

Ring 0 is mostly unnecessary on Apple Silicon

Apple already removed the attack surface a kernel driver exists to police. What remains is delivered through two Apple-supported user-space channels.

Windows threat Why it needs ring 0 on Windows macOS structural equivalent — no ring 0 needed
Memory read/write cheats OpenProcess requires handle strip via Ob callbacks in kernel Task-port model + GET_TASK ES events gate every access — catchable in user space
DLL/dylib injection AppInit_DLLs, remote thread — need image-load callbacks at ring 0 DYLD_INSERT_LIBRARIES caught at exec by ES; hardened runtime + library validation block unsigned dylibs at load
Unsigned kernel code (BYOVD) Kernel driver at ring 0 is the only thing that can police ring 0 No kexts without Apple notarization + explicit user approval + reboot; BYOVD is a non-path
DMA hardware cards Must verify VT-d/AMD-Vi IOMMU state at ring 0 DART IOMMU always on; no user-accessible PCIe on most Macs — commodity DMA cards largely don't apply
Boot-state integrity Relies on TPM (bolt-on, external to CPU) Secure Boot rooted in SEP (die-level, native) — attestable via App Attest without a driver
The trust inversion
Not "the client is unbreakable." The server only believes clients whose hardware attestation verifies. Tamper with the agent — its code signature changes, attestation breaks, and the server stops trusting it. Trust decisions live server-side, extended only to clients whose Apple-signed hardware proof verifies from scratch.
03 — Architecture

One client, two Apple channels

The security agent sits between the game engine and Apple's hardware trust infrastructure, reporting signed telemetry to a backend that makes all trust decisions.

system architecture — Vanguard-for-macOS
┌────────────────────────┐ │ GAME CLIENT / ENGINE │ └───────────┬────────────┘ │ in-process SDK ┌───────────▼────────────┐ OS ◄─┤ CLIENT SECURITY AGENT ├─► HARDWARE ES │ Phase 1: ES monitor │ SEP / App Attest events │ Phase 2: SEP + attest │ (Phase 2) └───────────┬────────────┘ │ signed telemetry + attestation ┌───────────▼────────────┐ │ BACKEND TELEMETRY SVR │ verify attestation vs Apple, │ trust: admit/flag/kick │ verify signed events, decide └────────────────────────┘ ───────────────────────────────────────────────────────────── ES layer → kernel-vetted event stream (exec, fork, task-port, dylib) SEP layer → non-exportable P-256 key; signs assertions in-enclave App Attest→ Apple certificate chain proves key is on genuine hardware Server → root of trust; verifies Apple cert chain from scratch
Pillar 1
Endpoint Security

Kernel-vended, user-space-consumed event stream. Subscribed event classes: EXEC, FORK, EXIT, GET_TASK, AUTH_GET_TASK, EXEC_ENV, LIBRARY_LOAD. Requires Apple-granted entitlement — the same gate every EDR vendor passes.

Pillar 2
Secure Enclave

P-256 key generated and stored on-die. Private key cannot be exported — not by the OS, not by root, not by the app. Signs assertions inside the enclave. Extraction would require physically attacking the chip.

Pillar 3
App Attest

Apple issues a certificate chain binding the SEP key to a specific appId + teamId on a specific physical device. A VM cannot produce a valid Apple-signed attestation. Replay-proof via server-issued challenges and a monotonic counter.

08 — Honest Gaps

What this does not solve

Stated up front — because a defense you oversell is a defense your red team stops trusting. These five gaps are out of scope by design, not by oversight.

  • 📊
    Server-side heuristics (aimbot input ML, ESP occlusion)
    Lives in the game engine and backend. This project secures the client trust anchor and local telemetry; it does not replace authoritative server design. The ES monitor gives the server ground truth about process state; the server must still decide what to do with it.
  • ⏱️
    Continuous runtime attestation
    App Attest is a trust anchor, not a per-frame heartbeat. It is rate-limited by Apple's servers by design. It proves the app was genuine at assertion time; it does not continuously prove the process is still clean. This is an architectural property of the platform, not a gap in the implementation.
  • 🐧
    Linux / SteamOS parity
    There is no Endpoint Security or App Attest equivalent on an open system where the user holds root. No Linux parity is claimed. This is the same boundary that keeps Vanguard off Linux — stated honestly rather than papered over.
  • 🖱️
    KMBox-class hardware input emulation
    A second PC running the aimbot, relaying input through a hardware USB/HID emulator. Best mitigated server-side via input-vector ML. Neither this project nor a kernel driver fully closes hardware input emulation client-side.
  • 🔒
    Fully compromised Apple platform (SEP key extraction)
    Out of scope for every commercial anti-cheat product. If an adversary can extract a non-exportable SEP key or forge an Apple-signed boot chain, the entire platform security model is void — and so is every competitor's.
09 — Comparison

vs. vgc.sys (Windows kernel driver)

Not "equivalent to ring 0." Ring 0 is mostly unnecessary on Apple Silicon — and we can prove the client is genuine.

Dimension vgc.sys (Windows kernel) Vanguard-for-macOS
Privilege level Ring 0, loads at boot — before the game, always running User space, runs with the game. One-process crash, not a system crash
Memory access defense Ob handle-strip callbacks at ring 0 — deny OpenProcess to the game ES GET_TASK / AUTH_GET_TASK events — deny task_for_pid without a kernel driver
Injection defense Kernel image-load callbacks catch DLL injection attempts dyld exec-env scan + hardened runtime library validation — platform rejects unsigned dylibs before they load
DMA cards Verify VT-d/AMD-Vi IOMMU at ring 0; still a real threat on many configurations DART IOMMU always-on; no user PCIe on most Mac SKUs — commodity DMA cards largely don't apply
Foreign kernel code BYOVD is an actively exploited live threat — signed vulnerable drivers are sold commercially No kexts without Apple notarization + explicit user approval + reboot — the attack surface barely exists
Trust anchor TPM (bolt-on, external chip, separate supply chain) SEP (die-level, native to Apple Silicon, same chip as CPU) + App Attest cert chain
Stability blast radius BSOD-class. Boot loops possible. Third-party driver bugs have triggered mass outages One user-process crash. Cannot panic the kernel. Cannot cause a boot loop
Distribution & vetting Studio ships and maintains the kernel driver, handles update/signing supply chain Apple-vetted entitlement — same review process every EDR/MDM vendor goes through. Apple maintains driver compatibility
The bottom line
The worst hardware cheat vector on Windows — DMA cards — is largely a non-threat on Mac out of the box. The hardest driver-security problem on Windows — BYOVD — barely exists as an attack path on macOS. The trust anchor the driver bolts on via TPM — macOS has natively at the silicon level. What a kernel driver adds on Windows, Apple has mostly removed the need for on macOS.
10 — Next Steps

For a reviewing studio

Concrete, sequenced steps for a security team evaluating this for production.

  1. Run Phase 1 on a test Mac. Attempt task_for_pid and DYLD_INSERT_LIBRARIES against a protected target process, confirm detection using the included test/demo_detections.sh harness. No entitlement required if SIP is disabled on the test machine.
  2. Review the Phase 2 server verifier (verify.js). Every Apple-spec check is commented and traceable to Apple's App Attest specification. Run npm test to confirm the five security properties hold. No Mac required — runs on any Node 18+ host.
  3. Apply for the entitlements. The ES entitlement (com.apple.developer.endpoint-security.client) and the App Attest macOS entitlement use the same Apple review process every EDR and MDM vendor goes through. For a studio of Riot's size this is a process cost, not a blocker.
  4. Scope the engine integration. Next concrete steps: wrap the ES monitor as a notarized System Extension (packaging only, no logic change); wire Phase 1 telemetry into the Phase 2 clientData envelope so each hardware-signed assertion carries the ES event log; integrate the server verifier into the game backend's trust decisions.
  5. Pressure-test the honest gaps. The project concedes server-side heuristics, continuous attestation, Linux parity, KMBox, and full platform compromise. If the security team identifies gaps beyond these, we want to know — a defense you oversell is a defense a red team stops trusting.
